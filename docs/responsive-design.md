# 멀티 디바이스 대응 (데스크톱 · 모바일 · 태블릿)

여러 화면 크기·화면비·입력 방식에서 **일관되고 고급스럽게** 동작하기 위한 이론과 dori 적용 규칙.

> **상위 원칙**: [design-principles.md](./design-principles.md)(어포던스)를 우선한다.
> 특히 **입력 방식(터치/마우스/스타일러스)에 맞는 정직한 조작·피드백**이 핵심이다.
> 시각 마감 값은 [visual-polish.md](./visual-polish.md), 스케일 설정은 [development.md](./development.md) 참고.

---

## 1. 반응형 vs 적응형

- **반응형(Responsive)**: 하나의 레이아웃이 화면 크기에 따라 유연하게 늘어남(유동 단위·비율).
- **적응형(Adaptive)**: 특정 브레이크포인트마다 다른 레이아웃을 전환.
- **dori 전략**: 게임 캔버스는 **반응형 스케일**(하나의 기준 해상도를 비율 유지하며 축소/확대), UI 배치는 필요 시 화면비에 따라 **적응형**으로 조정.

## 2. 모바일 퍼스트 + 유연한 레이아웃

- 가장 제약이 큰 **모바일 세로**를 기준으로 설계하고, 태블릿·데스크톱은 여백/스케일을 확장.
- 고정 px 좌표 대신 **화면 크기 기준 상대 배치**(`this.scale.width * 0.5` 등)를 사용.

## 3. 화면비/해상도 다양성

| 기기 | 대표 화면비 | 특징 |
|------|-------------|------|
| 모바일(세로) | 19.5:9 – 20:9 | 길쭉함, 노치/홈 인디케이터 |
| 태블릿 | 4:3 – 16:10 | 정사각에 가까움, 세로/가로 전환 |
| 데스크톱 | 16:9 – 21:9 | 넓음, 마우스/키보드 |

→ **하나의 해상도로 모두 맞출 수 없음.** 기준 해상도 + 스케일 + 안전 영역으로 대응한다.

## 4. Phaser Scale Manager

| 모드 | 동작 | 장단점 |
|------|------|--------|
| `FIT` | 비율 유지하며 화면에 맞춤 | 안전하지만 남는 곳에 여백(레터박스) |
| `ENVELOP` | 비율 유지하며 화면을 꽉 채움 | 여백 없음, 가장자리 잘림 |
| `RESIZE` | 캔버스를 화면 크기에 맞게 리사이즈 | 완전 반응형, 레이아웃 직접 재배치 필요 |

**dori 선택(현재):** `FIT` + `CENTER_BOTH` (기준 720×1280).
- **개선 방향**: 핵심 UI는 중앙 **안전 설계 영역(720×1280)** 안에 두고, **배경 아트를 바깥까지 확장**해 레터박스가 "빈 검은 띠"로 보이지 않게 한다(체감 품질 ↑). 넓은 화면비가 중요해지면 `RESIZE`로 전환해 요소를 재배치.

## 5. 안전 영역(Safe Area) & 뷰포트

- 노치·둥근 모서리·홈 인디케이터에 UI가 가리지 않도록 **안전 영역**을 확보.
- HTML: `<meta name="viewport" ... viewport-fit=cover>` + CSS `env(safe-area-inset-*)`.
- **dori 규칙**: 버튼·점수 등 **중요 UI는 화면 가장자리에서 최소 24px(안전 영역) 안쪽**에 배치.
- **뷰포트 높이는 `100dvh`**(폴백 `100vh`): 모바일 `100vh`는 브라우저 툴바 뒤 영역까지 포함하는 최대 높이라,
  하단 툴바형 브라우저(웨일 등)에서 캔버스 하단(허브 하단 바)이 툴바에 가려 잘렸다.
  `dvh`는 실제 보이는 높이를 따라 변하고, Phaser ScaleManager가 리사이즈를 감지해 다시 맞춘다.

## 6. 입력 방식 차이 (어포던스와 직결)

| 입력 | 특징 | 규칙 |
|------|------|------|
| 터치 | **hover 없음**, 손가락이 큼 | hover에만 의존하는 힌트 금지, 큰 타깃 |
| 마우스 | hover·정밀 클릭·커서 | hover는 **보조 강화**로만(없어도 동작) |
| 스타일러스/트랙패드 | 정밀, hover 가변 | 마우스에 준함 |

- **핵심**: hover 상태(버튼 밝아짐 등)는 **있으면 좋은 보너스**일 뿐, 조작 가능함은 **정적인 시그니파이어**(입체감·라벨)로 항상 드러낸다([design-principles.md](./design-principles.md)).
- 데스크톱: `useHandCursor: true`, 키보드(Enter/Space) 보조 지원 고려.

## 7. Fitts의 법칙 & 터치 타깃

- **Fitts의 법칙**: 타깃이 크고 가까울수록 빠르고 정확하게 누른다.
- 최소 터치 타깃: **44px(Apple) / 48dp(Google)** 물리 크기.
- **dori 환산 규칙**: 기준 720px 폭이 작은 폰(약 400px)에서 –0.55배로 축소되므로, **설계 단위로 터치 타깃 ≥ 88px, 타깃 간 간격 ≥ 16px**로 잡으면 모든 기기에서 편하다.

## 8. 엄지 영역(Thumb Zone) / 도달성

- 세로 모바일에서 한 손 조작 시 **화면 하단·중앙**이 닿기 쉽다.
- **dori 규칙**: 주요 액션 버튼(시작·다시하기·누르기)은 **하단–중앙**에, 정보 표시(점수·타이틀)는 상단에.

## 9. 방향(Orientation)

- **기본: 세로(portrait) 우선.** 파티/복불복 게임은 세로가 자연스럽다.
- 태블릿/데스크톱 가로에서는 **세로 캔버스를 중앙 정렬 + 배경으로 채움**(레터박스 대신 아트).
- 가로를 정식 지원할 경우 §4 `RESIZE`로 레이아웃 재배치.

## 10. 해상도 선명도 (DPR / Retina)

- 고해상도 디스플레이에서 흐릿하지 않게: Phaser는 기본 처리하지만, 코드 생성 텍스처/폰트는 **충분한 원본 크기**로 만들어 축소.
- 도트 아트가 아니면 `pixelArt: false`(안티앨리어싱 유지), 도트면 `true`.

---

## dori 적용 규칙 요약

- 기준 해상도 720×1280(세로), `FIT` + `CENTER_BOTH`, 배경은 바깥까지 확장.
- 좌표는 **상대 배치**(`scale.width/height` 기준).
- 터치 타깃 **≥ 88px(설계 단위)**, 간격 ≥ 16px, 중요 UI는 가장자리 **24px 안쪽**.
- hover는 **보조**, 조작 가능함은 항상 정적 시그니파이어로.
- 주요 버튼은 **하단–중앙**, 정보는 상단.
- 세로 우선, 가로는 배경 확장으로 대응.

## 체크리스트

- [ ] 여러 화면비(19.5:9 / 4:3 / 16:9)에서 UI가 안 잘리는가
- [ ] 레터박스가 빈 띠가 아니라 배경으로 채워지는가
- [ ] 노치/홈바에 중요 UI가 안 가리는가(안전 영역)
- [ ] 터치 타깃이 작은 폰에서도 충분히 큰가(≥88px 설계 단위)
- [ ] hover 없이도 조작 가능함이 보이는가(터치 대응)
- [ ] 주요 버튼이 엄지로 닿는 위치인가
- [ ] 고해상도에서 흐릿하지 않은가
