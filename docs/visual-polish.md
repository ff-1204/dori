# 시각 마감 / 체감 품질 (안 싼티 나게)

정적 호스팅(github.io)에서도 **고급스럽게 보이기 위한 이론과 dori 적용 규칙**을 정리한다.
github.io의 한계는 런타임이 아니라 **로딩(에셋 크기)** 뿐이며, "싼티"는 성능이 아니라 **연출·비주얼 마감** 문제다.

> **상위 원칙**: 이 문서는 [design-principles.md](./design-principles.md)(게이미피케이션·어포던스)의 하위 실천 지침이다.
> 마감이 화려해도 정직한 피드백·정직한 매핑을 해치면 안 된다.

---

## 1. 미학-사용성 효과 (Aesthetic-Usability Effect)

"예뻐 보이면 실제로 더 잘 작동한다고 느낀다." → 첫인상(타이틀·색·폰트)에 투자하면 게임 전체 평가가 관대해진다.

**적용:** 타이틀 화면·첫 진입 연출에 가장 많은 마감 예산을 쓴다.

## 2. 게임 필 / 주스(Juice)

같은 기능도 반응이 살아있으면 고급스럽다. 자세한 이론은 [game-theory.md](./game-theory.md) §5–6.

**적용 규칙:**
- 모든 입력에 즉각 반응(눌림 스케일 `0.94`, 복귀 `Back.easeOut`).
- 상태 전이·등장은 항상 트윈으로(팝: `scale 0→1`, `Back.easeOut`, 150ms).
- 파티클·화면 흔들림은 **절제**해서 결정적 순간에만.
- **60fps 유지**가 최우선 품질 신호. 무거운 연출보다 부드러움.

## 3. 시각 디자인 기초 (일관성이 곧 고급)

싼티의 대부분은 **정렬 안 맞음 + 색 과함 + 기본 폰트**에서 나온다.

### 3-1. 색 — 제한된 팔레트
- **60-30-10 규칙**: 배경 60% / 주요 30% / 강조 10%.
- 원색 남발 금지, 채도 통제. 아래 기본 팔레트 사용:

| 역할 | HEX | 용도 |
|------|-----|------|
| 배경 | `#12131c` | 앱 배경 |
| 표면 | `#1d1f2b` | 카드/패널 |
| 텍스트 | `#f2f3f7` | 본문 |
| 서브텍스트 | `#8b90a8` | 보조 |
| 강조(Primary) | `#6cc7ff` | 버튼/포인트 |
| 성공 | `#06d6a0` | 당첨/성공 |
| 경고 | `#ffd166` | 주의 |
| 위험 | `#ff6b6b` | 꽝/패배 |

- **참가자 식별색(`PLAYER`, 다인 게임 최대 6명)**: 강조 4색 + `#c77dff`(보라) + `#ff9f6c`(오렌지). 색상 연결(§3-1a) 전용 — 의미색과 혼동될 맥락에선 라벨 병행.

### 3-2. 간격 — 8pt 그리드
- 여백/간격은 **8의 배수**(8·16·24·32·48·64)로 통일. 빽빽하면 싼티, 여백이 고급.

### 3-3. 타이포그래피 위계
- 기본 sans 대신 **웹폰트 하나**를 골라 위계를 준다(예: Pretendard, Noto Sans KR).
- 스케일 예시(px): 타이틀 96 / 헤딩 52 / 본문 32 / 캡션 24.
- 한 화면에 폰트는 **1종**, 굵기로만 위계.

### 3-1a. 색상 연결 (Color Linkage) — 표준 기법
- **결과·피드백의 색을 그 출처 요소의 색과 매칭**한다. 예: 룰렛 당첨 칸이 노란색이면 "오늘 저녁은 ○○!" 텍스트·화면 플래시도 노란색.
- 효과: 결과가 어디서 왔는지 색으로 이어져 **정직한 매핑(어포던스)** 이 강화되고, 화면이 하나의 시스템처럼 느껴진다.
- 적용 대상: 결과 텍스트, 플래시/파티클, 하이라이트 등 **출처가 있는 모든 피드백**. 새 게임에도 기본 적용한다.
- 주의: 의미색(성공/위험)과 혼동될 수 있는 맥락에서는 라벨·아이콘을 병행(색만으로 의미 전달 금지).

### 3-4. 형태 & 정렬
- 라운드 코너 일관(예: 반경 16px), 그림자는 은은하게 1단계.
- **하나의 아트 디렉션**을 끝까지: 스타일 짜깁기가 가장 싼티하다.
- 게슈탈트(근접·유사·정렬)로 그룹을 명확히, 시각적 위계로 중요한 걸 크게.

## 4. 애니메이션 원칙 (모션 = 프리미엄감)

디즈니 12원칙 중 **이징·기대감(anticipation)·후속동작(follow-through)**.

**dori 이징 기본값:**
| 상황 | ease | duration |
|------|------|----------|
| 등장/팝 | `Back.easeOut` | 150ms |
| 퇴장 | `Back.easeIn` | 150ms |
| 룰렛/감속 | `Cubic.easeOut` | 3000–4000ms |
| 통통 마무리 | `Bounce.easeOut` | 400ms |
| 일반 이동 | `Quad.easeInOut` | 200–300ms |

정적인 화면보다 **살짝씩 움직이는 화면**이 훨씬 비싸 보인다.

## 5. 지각된 성능 (Perceived Performance) — github.io 로딩 대응

- **Doherty Threshold(400ms)**: 입력 반응은 즉각적으로. 지연되면 로딩 인디케이터.
- **로딩 연출**: 로딩 바/스켈레톤으로 체감 지연을 숨긴다(현재 `PreloadScene`).
- **에셋 최적화**(첫 로딩만 관리하면 됨):
  - 이미지 압축(WebP/PNG 최적화), 불필요 해상도 지양.
  - **스프라이트 아틀라스**로 요청 수 최소화.
  - CDN 캐싱 활용(Phaser는 CDN, 정적 파일은 GitHub Pages 캐시).
  - 가능하면 텍스처를 코드 생성(현재 타일 방식)해 에셋 자체를 줄인다.

## 체크리스트 (릴리스 전 "싼티" 점검)

- [ ] **요소끼리 겹치지 않는가** — 텍스트·포인터·장식의 y/x 점유 범위를 좌표로 계산해 확인(예: 포인터 y214~266이면 제목은 그 위 여백 확보)
- [ ] 첫 화면이 예쁜가(미학-사용성 효과)
- [ ] 모든 클릭/탭에 즉각 반응이 있는가(주스)
- [ ] 색이 팔레트 안에서만 쓰였는가(과한 원색 없음)
- [ ] 간격이 8pt 그리드로 정렬됐는가
- [ ] 폰트 위계가 명확한가(웹폰트·굵기)
- [ ] 화면 전환/등장에 이징 트윈이 있는가
- [ ] 60fps로 부드러운가
- [ ] 로딩이 비어 보이지 않는가(로딩 연출)
